Documentation 2

 

 

 

This part contains two files :

                                      1.task1.cc

                                      2.task2.h (note:task2.h contains both task 2 and task3 )

 

task1.cc  - contains functions to convert an infix to postfix.

task2.h  - header file, contains functions to build a parse tree , and traverse it to generate back the infix through  inorder traversal.

 

The code has been functionally modularized into consistent logical units,and C++ has  been the choice of language to reap the benefits of STL hence enhancing code reusability.

 

Note: g++  6.2.0  or any version above  it is recommended.

 

NOTE:

OUTPUT      

 

1.MATCHING - indicates given infix expression, and infix after traversing the generated  postfix are matching, thus verifying that the given formula is well formed .

2.NOT MATCHING - indicates, no match between given infix and generated infix after traversing the formed parse tree, thus an INVALID formula or NOT FULLY PARENTHESISED formula.

3.Time is given in the output is in seconds.

Time complexity

Time complexity of task1(infix to postfix conversion) : O(n*n)

Time complexity of task 2 (constructing a parse tree): O(n)

Time complexity of task 3 (traversing parse tree): O(2^n)

 

BREAKING POINT

·        If infix is given more than 232 - 1 or 4294967295 characters then the code fails .

·        Since a recursion is involved for very large inputs(depends on the size of stackframe allocated ) the code might fail to give the desired output.

 

ASSUMPTIONS

1.     A valid, well parenthesised formula is expected.

 

2.      If there is a mismatch in parenthesis an erroneous output/segmentation fault may result.

 

3.      The symbols to be used are

·        AND -  ^

·        OR - V

·        IMPLICATION- >

·        NEGATION- ~

 

    4.   If p is a literal, p is valid but (p) is not valid .

 

    5.If operand is a string of more than one character then infix after traversing the resulting postfix will not match with the given infix,indicating an invalid  input.

    Ex: (abVc)

 

           *6. ~ should always be followed by ( or a literal.

                Ex:  ~c is accepted, but ~(c) and ~~c  is given as input then infix after traversing the resulting  postfix will not match with the given infix, indicating an invalid input.

 

   **7. If input is not well parenthesised, then given the formula is valid, a correct postfix is  returned,but still NOT MATCHING is given as output since                while traversing the postfix the  resulting infix2 is well parenthesised.
